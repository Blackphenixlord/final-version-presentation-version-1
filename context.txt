PROJECT OVERVIEW
----------------
Full-stack inventory management system for NASA's Deep Space Logistics Module
(DSLM). Built under the NASA HUNCH program. Tracks supplies from external
vendor → KSC Ground warehouse → ISS Crew. Three role-based UIs share a common
Fastify backend and RFID scanning subsystem.

SUPPLY CHAIN FLOW
-----------------
  1. Ground places Purchase Order → vendor sees it on My Orders (polled 5s)
  2. Vendor picks items, tags each with RFID, ships → invoice auto-generated
  3. Ground receives inbound shipment (ReceiveScreen), counts line items
  4. Ground tags remaining items (TagScreen)
  5. Ground packs into CTBs (PackScreen)
  6. Ground stows in 3D module / module grid (StowScreen / ModuleView3D)
  7. Items launch to ISS
  8. Crew takes out / returns / disposes via RFID scan
  9. Crew sends Low-Resources request → Ground approves / fulfils

CTB SPECIFICATIONS (NASA standard)
----------------------------------
  0.5 CTB:  42.5 × 24.8 × 23.5 cm  (7.3 kg max,  0.025 m³)
  1.0 CTB:  50.2 × 42.5 × 24.8 cm  (15.4 kg max, 0.053 m³)
  2.0 CTB:  50.2 × 42.5 × 50.2 cm  (31.0 kg max, 0.107 m³)
  3.0 CTB:  74.9 × 42.5 × 50.2 cm  (46.4 kg max, 0.160 m³)
  4.0 CTB:  89.7 × 50.8 × 53.3 cm  (70.5 kg max, 0.243 m³)
  6.0 CTB:  89.7 × 81.9 × 53.3 cm  (113.7 kg max, 0.392 m³)
  CTBs can nest inside each other. No bracing between CTBs in stacks.

RFID SYSTEM
-----------
  Hardware:   UHF RFID reader connected via USB serial (keyboard-wedge mode).
              The reader emulates keyboard input; the useKeyboardWedgeScan hook
              debounces 120ms and auto-submits.
  Simulation: Type badge/tag UID in any scan input field and press Enter.
  Tag spec:   EPC Gen 2, 860–960 MHz. Avoid placement within 2 cm of metal.
  Baud/port:  Configure in vite.config.ts proxy or via serialport at the server.
              On Linux (Jetson/Pi), typical path: /dev/ttyUSB0 at 9600 baud.
  Badge UIDs: crew=0003070837  ground=0003104127  vendor=0003200001

X-400 / X-500 MESSAGES
-----------------------
  Ground Dashboard polls /api/messages every 4 s.
  X-400: standard operations messages (disposal, stow events).
  X-500: high-priority or ISS-to-Ground urgent traffic.
  Source: crew dispose action auto-sends an X-400 message.

3D MODULE VIEW
--------------
  File: nasa-hunch/src/screens/ModuleView3D.tsx
  Library: Three.js via @react-three/fiber + @react-three/drei

  Geometry: Real DSLM — 3.7 m diameter × 4 m cylinder (module runs along X-axis)
    S-1  top center           — rectangular prism 0.85 m × 1.0 m × 4 m
    S-2  lower-right          — same
    S-3  lower-left           — same
    C1   upper-left column    — curved-back, volume ~2.56 m³ (non-interactive)
    C2   upper-right column   — curved-back, volume ~2.56 m³ (non-interactive)
    S-4  under floor          — NASA standard lockers (non-interactive)
    Translation pathway clear: 1.143 m × 0.813 m through center

  Slot grid per stack: 4 depths (D1–D4, 1 m each) × 12 levels (4 cols × 3 rows)
                       = 144 slots per stack, 432 total
  Features:
    - Click empty slot → StowPanel popup → enter item ID → POST /api/stow
    - Double-click occupied slot → container tree (GET /api/containers/:id/tree)
    - Slot search/filter, fresh-pulse animation on recent changes
    - ReceiveDock (green, x=-8.5) and TrashZone (red, x=+8.5) platforms
    - Activity feed (color-coded log events from /api/logs)
    - Theme-aware colours (dark/light NORD palettes)
    - Real-time polling: locations 4 s, logs 5 s, inbound count 8 s
    - mode prop ("crew" | "ground") controls available actions
    - AutoRotate when idle

BACKEND IN-MEMORY SEED DATA (NO_DB=1)
--------------------------------------
  Badges:  0003070837 → crew, 0003104127 → ground, 0003200001 → vendor
  Grid:    3 shelves × 4 depths × 12 levels = 144 stow slots
  Items:   ~30 seeded inventory items (medical, food, tools, hygiene, maintenance)
  RFID:    288 pre-seeded RFID items (188 Cards + 100 Fobs)
           Added in server.mjs after createMemoryStore() unit loop.
           SEED_CARDS and SEED_FOBS arrays → seedRfidItem() populates
           units + identifiers maps with status="NEEDS_VERIFY".
           All 288 appear in the Tag screen as "needs-verify" on server start.
  No persistence — all state resets on server restart. Use PostgreSQL for
  persistence (set DATABASE_URL in .env and remove NO_DB=1).

ENVIRONMENT VARIABLES
---------------------
  NO_DB=1          Use in-memory store (no PostgreSQL required)
  DATABASE_URL     PostgreSQL connection string (optional)
  CORS_ORIGINS     Comma-separated allowed origins (default: "*")
  PORT             Server port (default: 8080)

ELECTRON / WINDOWS EXE (added 2026-04-28)
-------------------------------------------

  Output:  nasa-hunch/dist-electron/win-unpacked/DSLM NASA HUNCH.exe  (223 MB)
  Zip:     nasa-hunch/dist-electron/DSLM-NASA-HUNCH-win-x64.zip  (177 MB)

  How it works:
    1. npm run build:win  →  npm run build + electron-builder --win --x64
    2. main.cjs spawns the Fastify backend (node server.mjs, NO_DB=1)
    3. After 800ms delay, opens BrowserWindow loading dist/index.html
    4. Backend embedded via extraResources in electron-builder config

  Note: NSIS installer requires Windows Developer Mode (symlink privilege).
        Use zip/dir targets (already configured) to avoid this requirement.
